Памятка по асинхронности в Godot 4 (GDScript)

Объявление асинхронной функции: не используй ключевое слово async. Любая функция, внутри которой есть хотя бы один await, автоматически становится асинхронной.
Пример:

gdscript
func do_something():
    await get_tree().create_timer(1.0).timeout
    print("Done")
Вызов асинхронной функции: чтобы дождаться завершения, используй await перед вызовом:

gdscript
await do_something()
Если await не поставить, функция запустится, но продолжение кода не будет ждать её окончания (выполнение продолжится сразу). Это может привести к состояниям гонки, если нужен результат.

Возврат значений: асинхронная функция может возвращать значение (например, func get_data() -> int:), и тогда после await можно получить результат. Но если функция не возвращает явно (возвращает void), после await получишь null.

Использование await с сигналами: await работает с любым сигналом:

gdscript
await some_object.some_signal
Обработчики сигналов: если в обработчике сигнала нужно выполнить асинхронный код (например, вызвать await), просто используй await внутри обработчика. Сигнал не ждёт завершения обработчика, поэтому будь внимателен при последовательных асинхронных операциях — они могут выполняться параллельно. При необходимости используй флаги или блокировки.

Где нельзя использовать await: в функциях, которые должны выполняться синхронно (например, _ready(), _process()), await технически допустим, но может привести к неожиданностям, так как основной цикл продолжит выполняться. Лучше запускать асинхронные задачи через call_deferred() или использовать корутины.

Наследование: при переопределении асинхронного метода (например, func execute(context) -> void: в классе Ability) в наследнике не пиши async, но внутри можешь использовать await. Контракт метода (сигнатура) остаётся тем же.

Ошибка "Unexpected identifier 'async'": возникает, если написать async func в GDScript. Этого ключевого слова в языке нет, просто не используй его.

Краткий пример:

gdscript
# Правильно
func load_data():
    print("Loading...")
    await get_tree().create_timer(2.0).timeout
    print("Loaded")
    return 42

# Вызов с ожиданием результата
var result = await load_data()
print(result)  # 42

# Памятка по разработке игры в Godot 4 (GDScript)

## 1. Асинхронность
- Не пиши `async` перед функцией – достаточно использовать `await` внутри.
- Обработчики сигналов не могут быть `async`, но внутри них можно вызывать `await` (сигнал не ждёт завершения, но это допустимо).
- При вызове асинхронной функции всегда используй `await`, иначе она выполнится без ожидания.

## 2. Инициализация узлов (@onready)
- Переменные с `@onready` инициализируются **после** добавления узла в дерево.
- Если создаёшь экземпляр сцены через `instantiate()`, сначала добавь его в дерево (`add_child`), затем вызывай `setup()`. Иначе `@onready` переменные будут `null`.

## 3. Поиск элементов в стеке
- Для манипуляций со стеком (перемещение элементов) используй уникальный идентификатор (`id`) в каждом элементе.
- Не полагайся на сравнение словарей через `find` – они могут не совпадать. Лучше хранить `id` и искать по нему.

## 4. Разрешение стека
- **Пошаговое разрешение:** каждая способность выполняется только после нажатия кнопки «Разрешить» (или автоматически после выбора цели). Это даёт игроку контроль над порядком.
- После выполнения Upheaval (или другой способности, добавляющей новые эффекты) стек обновляется, но разрешение **не продолжается автоматически** – игрок должен снова нажать кнопку.
- При запросе цели разрешение приостанавливается; после выбора цели способность выполняется, и стек останавливается (ждёт следующего нажатия).

## 5. Выбор цели
- Цель должна запрашиваться **в момент разрешения способности**, а не при добавлении в стек. Это соответствует MTG.
- Во время выбора цели блокируй кнопку «Разрешить» и другие элементы управления стеком, чтобы избежать конфликтов.

## 6. Дублирование вызовов способностей
- Не испускай дважды один и тот же сигнал (например, `triggered` и `EventBus.gear_triggered`). В `gear.gd` в `trigger()` оставь только `triggered.emit(self)`.
- Все триггерные способности должны добавляться в стек через `StackManager`, а не выполняться немедленно.

## 7. Анимации (Tween)
- Перед созданием нового `Tween` убивай старый, если он ещё активен, чтобы избежать блокировки:
  ```gdscript
  if _current_tween and _current_tween.is_valid():
      _current_tween.kill()
После kill можно создавать новый и ожидать его завершения.

8. Структура проекта
Разделяй ответственность: GameManager не должен содержать всю логику. Используй компоненты:

GameInitializer – загрузка данных, создание игроков.

GameRuleValidator – проверки правил.

GameEventHandler – обработка сигналов от объектов.

GameUIManager – обновление UI через события.

StackManager – управление стеком способностей.

9. Отвязка UI от логики
UI не должен знать о GameManager. Вместо этого:

В GameUIManager подпишись на события EventBus и обновляй UI методами.

В способностях и командах не вызывай update_ui() напрямую – изменения произойдут по событиям.

10. Типичные ошибки
Забыл добавить узел в дерево перед вызовом setup → @onready переменные null.

Использовал find по словарю вместо сравнения по id.

Не убил старый Tween → анимация не запускается.

Двойная эмиссия сигнала → способность выполняется дважды.

Автоматическое разрешение стека после добавления новых эффектов (например, после Upheaval) → игрок не успевает изменить порядок.

11. Порядок добавления в стек
Используй push_front, чтобы последний добавленный был первым в очереди (LIFO).

При отображении в UI показывай элементы в том порядке, в котором они будут выполняться (первый сверху – следующий на выполнение).

12. Валидация целей
При запросе цели проверяй, что цель всё ещё существует и легальна (в момент разрешения). Если цель исчезла, способность должна провалиться (как в MTG).

13. Рефакторинг
Постепенно убирай прямые зависимости от GameManager в способностях и командах. Внедряй через интерфейсы или передавай только необходимые компоненты.

14. Полезные сигналы в EventBus
stack_updated – для обновления UI стека.

stack_resolved – для оповещения, что стек опустел (используется в ResolutionPhase для перехода к следующей G).

target_selection_started – для блокировки кнопок во время выбора цели.

15. Отладка
Включай логирование ключевых моментов (добавление в стек, разрешение, выбор цели) – это помогает быстро находить ошибки.

При проблемах с отображением проверяй, не перекрывают ли элементы друг друга (используй mouse_filter, z_index, временные цвета фона).
