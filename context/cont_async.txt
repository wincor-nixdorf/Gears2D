Памятка по разработке игры Gears (Godot 4)
1. Асинхронность и ожидание сигналов
1.1. Основные правила
В GDScript нет ключевого слова async. Функция становится асинхронной автоматически при наличии await внутри.

При вызове асинхронной функции всегда используй await, иначе выполнение продолжится без ожидания.

После await всегда проверяй существование объекта: if not is_instance_valid(obj): return

1.2. Ожидание выбора цели

        await _wait_for_next_frame()

2. Interrupt механика
2.1. Последовательность действий при Interrupt
В фазе Resolution, когда в стеке есть способность, открывается окно Interrupt для неактивного игрока

Игрок выбирает Interrupt из руки, затем кликает на клетку (ортогонально смежную с разрешаемой)

Interrupt размещается на доске как AG (Active Gear) и становится последним звеном цепочки

Стек очищается

Разрешение цепочки перезапускается с Interrupt

2.2. Ключевые методы
_run_interrupt_window_before_resolve() — открывает окно для Interrupt

_play_interrupt() — размещает Interrupt на доске

set_interrupt_played() — устанавливает состояние Interrupt

restart_with_interrupt() — перезапускает разрешение с Interrupt

4. Фаза разрешения цепочки (ResolutionPhase)
4.1. Структура шагов
BEFORE — до разрешения (приоритет)

RESOLVE — авто-тик, флип, триггеры, обработка стека

AFTER — после разрешения (приоритет)

4.2. Обработка стека и Interrupt
gdscript
func _handle_stack_and_interrupts() -> void:
    while not stack_empty:
        # Ждём выбора цели если нужно
        if stack_manager.has_pending_target():
            await _wait_for_target_selection()
            continue
        
        # Открываем окно Interrupt
        await _run_interrupt_window_before_resolve()
        
        if interrupt_played:
            # Добавляем Interrupt в начало очереди
            _resolve_queue.insert(0, _interrupt_pos)
            _current_index = 0
            return  # Выход для перезапуска
        
        # Разрешаем следующий элемент стека
        stack_manager.resolve_next()
        await _wait_for_stack_or_interrupt()

5. Выбор цели для способностей
5.1. Механика выбора
Способность с target_type != NO_TARGET запрашивает цель через push_effect_with_target()

UI подсвечивает возможные цели

После выбора цель сохраняется в StackEntry

func get_possible_targets(context: Dictionary) -> Array:
    var result = []
    # Добавляем игроков
    for player in game_manager.get_players():
        result.append(player)
    # Добавляем шестерни
    for gear in board_manager.get_all_gears():
        if gear == source: continue
        if gear.type == GameEnums.GearType.INTERRUPT: continue
        if not gear.is_face_up:
            result.append(gear)  # DG
        elif gear.type == GameEnums.GearType.CREATURE:
            result.append(gear)  # Существо
    return result

6.2. Метод has_prevent_trigger (одноразовый)

func has_prevent_trigger(target: Object) -> bool:
    if not _modifiers.has(target): return false
    for i in range(_modifiers[target].size() - 1, -1, -1):
        if _modifiers[target][i].tag == "prevent_trigger":
            _modifiers[target].remove_at(i)
            if _modifiers[target].is_empty():
                _modifiers.erase(target)
            return true
    return false

9. Типичные ошибки и их решения
9.1. После Interrupt не продолжается разрешение
Причина: неправильный выход из _handle_stack_and_interrupts()

Решение: после Interrupt выходим из метода, но не вызываем _resolve_next() — разрешение продолжится через _start_resolve()

9.2. Не отображается окно стека для способности Interrupt
Причина: не проверяется has_pending_target() перед открытием окна Interrupt

Решение: добавить проверку в _handle_stack_and_interrupts()

9.3. Spark может выбрать Interrupt как цель
Причина: Interrupt не исключается из get_possible_targets()

Решение: проверять gear.type == GameEnums.GearType.INTERRUPT и пропускать

9.4. Модификатор prevent_trigger не срабатывает
Причина: проверка has_modifier() не удаляет модификатор

Решение: использовать has_prevent_trigger() который потребляет модификатор
